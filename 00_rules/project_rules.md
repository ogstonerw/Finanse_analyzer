# Правила проекта

## Назначение
- Репозиторий используется для подготовки ВКР и разработки программного продукта.
- Тема проекта не меняется без прямого указания пользователя.

## Базовые правила работы
- Перед началом задачи читать актуальные правила и материалы по теме проекта.
- Использовать только согласованную пользователем структуру ВКР.
- Не подменять согласованную структуру ВКР типовой структурой методички.
- Не изменять файлы вне явно указанной задачи.
- Не переписывать проект целиком без прямого запроса.

## Правила по ВКР
- В текстах диплома писать кратко, технически точно и без воды.
- Не выдумывать данные, метрики, эксперименты, ссылки и источники.
- Все выводы и проектные решения привязывать к разрабатываемой платформе.
- Главы 1 и 2 должны обосновывать проектирование, реализацию и оценку платформы в главе 3.

## Согласованная структура ВКР
- Глава 1. Анализ предметной области и требований к платформе
  - 1.1. Особенности фондового рынка как объекта интеллектуального анализа
  - 1.2. Классификация новостных, макроэкономических и сырьевых факторов, влияющих на рынок
  - 1.3. Методы анализа тональности, извлечения событий и прогнозирования реакции рынка на финансовых данных
  - 1.4. Анализ существующих платформ и инструментов оценки влияния новостей на фондовый рынок
  - 1.5. Выработка рекомендаций к разработке облачной платформы
  - 1.6. Функциональные и нефункциональные требования к системе
- Глава 2. Проектирование архитектуры и алгоритмов платформы
  - 2.1. Структурная схема и общая архитектура облачной платформы
  - 2.2. Проектирование базы данных и структуры обработки данных
  - 2.3. Проектирование интеллектуального модуля анализа новостей и генерации сигналов
  - 2.4. Проектирование кризисометра и логики интегральной оценки состояния рынка
  - 2.5. Проектирование пользовательского интерфейса и API взаимодействия компонентов
- Глава 3. Разработка, тестирование и оценка платформы
  - 3.1. Реализация серверной части платформы на Go
  - 3.2. Реализация клиентской части платформы на React
  - 3.3. Реализация AI-модуля обработки новостей и генерации комбинированного сигнала
  - 3.4. Тестирование на исторических рыночных кейсах
  - 3.5. Оценка качества прогнозов и эксплуатационных характеристик системы

## Правила по продукту
- Не менять тему проекта, целевые активы, горизонт прогноза и базовую продуктовую архитектуру без прямого указания пользователя.
- Backend разрабатывать на Go.
- Frontend разрабатывать на React.
- В качестве БД использовать PostgreSQL.
- Историю прогнозов хранить обязательно.

## Правила по источникам
- Перед разработкой интеграции проверять официальные и устойчивые источники данных.
- Приоритет отдавать биржам, регуляторам, статистическим службам и центрам раскрытия информации.
- Все релевантные источники фиксировать в `04_materials/sources_registry.md`.
- Не использовать неофициальный источник как единственный канал данных.

## Правила по порядку выполнения
- Если задача касается архитектуры или интеграций, сначала обновлять документацию.
- Если задача касается новой сессии, после завершения обновлять соответствующий файл в `05_session_logs/`.
- Если задача касается нового источника данных, сначала обновлять реестр источников, затем описание коннекторов или код.

- Перечислить измененные файлы.

## Содержание результата
- Добавлена SQL-миграция для `technical_indicators` с upsert-совместимой уникальностью по `asset_id`, `timeframe`, `indicator_time`.
- Для MVP рассчитываются `RSI(14)`, недельная доходность, историческая волатильность, направление тренда и положение цены в локальном диапазоне.
- Если данных на конкретную дату недостаточно, в БД сохраняется понятный `calculation_status` без выдуманных значений в числовых полях.
- На старте backend после синхронизации цен выполняется расчет индикаторов по уже сохраненным дневным свечам.
- Исправлено преобразование nullable-значений индикаторов из БД, чтобы endpoint корректно возвращал `null` там, где данных для расчета пока недостаточно.

## Принятые решения
- В качестве таймфрейма MVP используется только `1d`.
- Расчет тренда выполнен через сравнение короткой и длинной скользящих средних по закрытиям.
- Историческая волатильность считается по дневным лог-доходностям.
- Положение цены в канале рассчитывается по локальному 20-дневному диапазону high/low.

## Учтенные ограничения
- Использованы только уже сохраненные дневные свечи из `price_candles`.
- Не реализовывались сложные торговые сигналы, уровни поддержки и сопротивления, машинное обучение и AI-интеграция.
- Изменения внесены только в разрешенные backend-каталоги, README и session log.

# Логическая схема базы данных платформы

## Назначение схемы данных

Логическая схема базы данных предназначена для хранения нормативно-справочной информации, первичных данных из открытых источников, результатов предобработки, вычисленных технических признаков, прогнозов и пользовательских данных. В рамках проектируемой платформы база данных должна обеспечивать согласованное взаимодействие между модулями сбора данных, интеллектуального анализа новостей, расчета недельного комбинированного сигнала, кризисометра и пользовательского интерфейса.

В качестве основного хранилища предполагается использование PostgreSQL. Выбор реляционной модели обусловлен необходимостью поддерживать устойчивые связи между сущностями, хранить историю расчетов и обеспечивать целостность аналитических данных.

## Общие принципы проектирования

При формировании логической схемы БД учитываются следующие принципы:

- разделение справочных, первичных и производных расчетных данных;
- хранение истории прогнозов и признаков без потери предыдущих значений;
- возможность повторного расчета аналитических результатов на основе сохраненных входных данных;
- поддержка связей между новостями, событиями, активами и прогнозами;
- обеспечение расширяемости по составу источников и финансовых инструментов.

## Состав основных таблиц

В минимальную логическую схему платформы включаются следующие таблицы:

- `users`;
- `assets`;
- `sources`;
- `news_items`;
- `events`;
- `price_candles`;
- `technical_indicators`;
- `forecasts`;
- `market_regime`;
- `user_sessions`.

Ниже приведено проектное описание каждой таблицы.

## Таблица `users`

### Назначение

Таблица предназначена для хранения учетных записей пользователей платформы. Она используется модулем авторизации, backend API и механизмом доступа к персональной истории работы с системой.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ пользователя. |
| `email` | `varchar(255)` | Уникальный адрес электронной почты для входа в систему. |
| `password_hash` | `varchar(255)` | Хэш пароля пользователя. |
| `full_name` | `varchar(255)` | Имя пользователя для идентификации в интерфейсе. |
| `role` | `varchar(50)` | Роль пользователя, например `user` или `admin`. |
| `is_active` | `boolean` | Признак активности учетной записи. |
| `created_at` | `timestamp` | Момент создания записи. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |
| `last_login_at` | `timestamp` | Время последнего успешного входа. |

### Связи с другими сущностями

- Один пользователь может иметь несколько записей в таблице `user_sessions`.
- Один пользователь может быть связан с несколькими записями в `forecasts`, если система хранит пользовательскую историю просмотров, сохранений или инициированных расчетов.

## Таблица `assets`

### Назначение

Таблица содержит справочную информацию о финансовых инструментах и внешних факторах, анализируемых платформой. Она используется как центральный справочник активов для рыночных данных, индикаторов, событий и прогнозов.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ актива. |
| `ticker` | `varchar(50)` | Тикер или краткий идентификатор инструмента. |
| `name` | `varchar(255)` | Полное наименование актива. |
| `asset_type` | `varchar(50)` | Тип сущности: акция, индекс, товарный фактор и др. |
| `sector` | `varchar(100)` | Отрасль или категория актива. |
| `currency` | `varchar(10)` | Валюта котирования. |
| `is_active` | `boolean` | Признак использования актива в системе. |
| `created_at` | `timestamp` | Момент добавления актива в справочник. |
| `updated_at` | `timestamp` | Момент последнего обновления карточки актива. |

### Связи с другими сущностями

- Один актив может иметь множество записей в `price_candles`.
- Один актив может иметь множество записей в `technical_indicators`.
- Один актив может быть связан со множеством записей в `events`.
- Один актив может иметь множество прогнозов в `forecasts`.
- Один актив может быть связан со множеством состояний рынка в `market_regime`, если режим рассчитывается на уровне конкретного инструмента.

## Таблица `sources`

### Назначение

Таблица предназначена для хранения сведений об источниках данных, используемых платформой. Она описывает происхождение новостей, рыночных данных и других входных потоков.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ источника. |
| `name` | `varchar(255)` | Название источника. |
| `base_url` | `text` | Базовый URL или адрес источника. |
| `source_type` | `varchar(50)` | Тип источника: новости, рынок, макроэкономика, сырье. |
| `access_method` | `varchar(50)` | Способ доступа: API, файл, RSS, web. |
| `status` | `varchar(50)` | Статус использования источника в системе. |
| `update_frequency` | `varchar(100)` | Периодичность обновления данных. |
| `last_checked_at` | `timestamp` | Время последней проверки доступности или актуальности. |
| `created_at` | `timestamp` | Момент регистрации источника. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Один источник может быть связан со многими записями в `news_items`.
- При дальнейшем расширении схемы таблица может использоваться и для связи с рыночными или макроэкономическими рядами.

## Таблица `news_items`

### Назначение

Таблица хранит исходные новостные материалы, загруженные из внешних источников. Она представляет первичный слой текстовых данных до их окончательной интерпретации в виде событий.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ новости. |
| `source_id` | `uuid` | Внешний ключ на таблицу `sources`. |
| `external_id` | `varchar(255)` | Идентификатор новости во внешнем источнике. |
| `title` | `text` | Заголовок новости. |
| `summary` | `text` | Краткое описание или аннотация, если доступна. |
| `content` | `text` | Полный текст новости после загрузки и очистки. |
| `published_at` | `timestamp` | Время публикации материала. |
| `language` | `varchar(20)` | Язык публикации. |
| `url` | `text` | Ссылка на оригинал материала. |
| `sentiment_label` | `varchar(50)` | Предварительная текстовая оценка тональности. |
| `processing_status` | `varchar(50)` | Стадия обработки новости в системе. |
| `created_at` | `timestamp` | Момент записи новости в базу. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Каждая новость относится к одному источнику из `sources`.
- Одна новость может быть связана с одним или несколькими записями в `events`, если в одном тексте выделяется несколько значимых событий.

## Таблица `events`

### Назначение

Таблица предназначена для хранения структурированных событий, выделенных из новостных материалов. Она является промежуточным аналитическим слоем между исходным текстом и прогнозным модулем.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ события. |
| `news_item_id` | `uuid` | Внешний ключ на таблицу `news_items`. |
| `asset_id` | `uuid` | Внешний ключ на таблицу `assets`, если событие относится к конкретному активу. |
| `event_type` | `varchar(100)` | Тип события: корпоративное, макроэкономическое, политическое и др. |
| `event_scope` | `varchar(50)` | Масштаб влияния: локальный, отраслевой, рыночный, системный. |
| `title_normalized` | `text` | Нормализованное краткое описание события. |
| `impact_direction` | `varchar(50)` | Предполагаемое направление влияния события. |
| `impact_strength` | `numeric(5,2)` | Оценка силы потенциального влияния на нормализованной шкале. |
| `confidence_score` | `numeric(5,2)` | Уровень уверенности в корректности извлечения и классификации события. |
| `event_time` | `timestamp` | Время, к которому относится событие. |
| `explanation` | `text` | Краткое текстовое описание сути события. |
| `created_at` | `timestamp` | Момент фиксации события в системе. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Каждое событие связано с одной новостью из `news_items`.
- Событие может быть связано с одним активом из `assets`, если воздействие адресовано конкретному инструменту.
- Одно событие может использоваться в нескольких записях `forecasts`, например при пересчете прогноза или формировании прогнозов по нескольким связанным инструментам.

## Таблица `price_candles`

### Назначение

Таблица хранит рыночные временные ряды в форме свечей и используется для построения технических признаков, анализа динамики актива и расчета рыночного режима.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ записи. |
| `asset_id` | `uuid` | Внешний ключ на таблицу `assets`. |
| `timeframe` | `varchar(20)` | Таймфрейм свечи, например `1d` или `1h`. |
| `candle_time` | `timestamp` | Временная метка свечи. |
| `open_price` | `numeric(18,6)` | Цена открытия. |
| `high_price` | `numeric(18,6)` | Максимальная цена периода. |
| `low_price` | `numeric(18,6)` | Минимальная цена периода. |
| `close_price` | `numeric(18,6)` | Цена закрытия. |
| `volume` | `numeric(20,6)` | Объем торгов за период. |
| `created_at` | `timestamp` | Момент загрузки записи в базу. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Каждая свеча относится к одному активу из `assets`.
- На основе записей `price_candles` рассчитываются строки в `technical_indicators`.
- Ряды `price_candles` используются для определения режима рынка в `market_regime`.

## Таблица `technical_indicators`

### Назначение

Таблица содержит рассчитанные технические и рыночные признаки, применяемые для формирования недельного сигнала и анализа состояния рынка.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ записи индикатора. |
| `asset_id` | `uuid` | Внешний ключ на таблицу `assets`. |
| `indicator_time` | `timestamp` | Момент, для которого рассчитаны признаки. |
| `timeframe` | `varchar(20)` | Таймфрейм расчета. |
| `weekly_return` | `numeric(10,4)` | Недельная доходность актива. |
| `short_trend` | `numeric(10,4)` | Числовая оценка краткосрочного тренда. |
| `medium_trend` | `numeric(10,4)` | Числовая оценка среднесрочного тренда. |
| `rsi` | `numeric(10,4)` | Значение RSI. |
| `volatility` | `numeric(10,4)` | Оценка волатильности. |
| `channel_position` | `numeric(10,4)` | Положение цены в локальном канале. |
| `support_distance` | `numeric(10,4)` | Расстояние до ближайшей поддержки. |
| `resistance_distance` | `numeric(10,4)` | Расстояние до ближайшего сопротивления. |
| `market_breadth_score` | `numeric(10,4)` | Оценка ширины рынка или относительной динамики. |
| `created_at` | `timestamp` | Момент сохранения набора признаков. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Каждый набор индикаторов относится к одному активу из `assets`.
- Значения из `technical_indicators` используются при формировании записей в `forecasts`.
- Часть признаков используется при расчете `market_regime`.

## Таблица `forecasts`

### Назначение

Таблица хранит результаты недельного комбинированного прогноза. Она является основной расчетной таблицей прикладного контура платформы и используется для выдачи текущих и исторических аналитических результатов.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ прогноза. |
| `asset_id` | `uuid` | Внешний ключ на таблицу `assets`. |
| `event_id` | `uuid` | Внешний ключ на таблицу `events`, если прогноз связан с конкретным событием. |
| `user_id` | `uuid` | Внешний ключ на таблицу `users`, если прогноз сохранен в пользовательской истории. |
| `forecast_horizon` | `varchar(50)` | Горизонт прогноза, например `1w`. |
| `forecast_time` | `timestamp` | Время формирования прогноза. |
| `direction_label` | `varchar(50)` | Направление сигнала. |
| `signal_strength` | `numeric(5,2)` | Сила сигнала на нормализованной шкале. |
| `confidence_score` | `numeric(5,2)` | Уровень уверенности прогноза. |
| `market_regime_id` | `uuid` | Ссылка на актуальный режим рынка из `market_regime`. |
| `crisis_level` | `numeric(5,2)` | Интегральное значение кризисометра, использованное в расчете. |
| `summary_explanation` | `text` | Краткое текстовое объяснение прогноза. |
| `key_factors_json` | `jsonb` | Структурированное описание ключевых факторов и их вкладов. |
| `created_at` | `timestamp` | Момент сохранения прогноза. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Каждый прогноз относится к одному активу из `assets`.
- Прогноз может быть связан с одним событием из `events`.
- Прогноз может быть связан с одним пользователем из `users`.
- Прогноз может ссылаться на запись режима рынка из `market_regime`.

## Таблица `market_regime`

### Назначение

Таблица предназначена для хранения агрегированной оценки рыночного режима, применяемой как контекстный фактор для генерации сигнала и интерпретации кризисометра. Она отражает общее состояние рынка либо конкретного актива на заданный момент времени.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ режима рынка. |
| `asset_id` | `uuid` | Внешний ключ на таблицу `assets`, если режим рассчитывается для конкретного актива. |
| `regime_time` | `timestamp` | Момент определения режима. |
| `regime_label` | `varchar(50)` | Качественная метка режима: стабильный, напряженный, кризисный и др. |
| `regime_score` | `numeric(5,2)` | Нормализованная числовая оценка режима. |
| `market_stress_score` | `numeric(5,2)` | Частная оценка рыночного стресса. |
| `news_stress_score` | `numeric(5,2)` | Частная оценка новостного стресса. |
| `macro_stress_score` | `numeric(5,2)` | Частная оценка макроэкономического стресса. |
| `commodity_stress_score` | `numeric(5,2)` | Частная оценка сырьевого стресса. |
| `breadth_stress_score` | `numeric(5,2)` | Частная оценка ширины рынка и поведения ключевых активов. |
| `explanation` | `text` | Краткая расшифровка доминирующих факторов режима. |
| `created_at` | `timestamp` | Момент сохранения записи. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Режим рынка может быть общерыночным или привязанным к одному активу из `assets`.
- Одна запись `market_regime` может использоваться во множестве прогнозов `forecasts`.
- Частные оценки режима могут рассчитываться на основе данных из `news_items`, `events`, `price_candles` и `technical_indicators`.

## Таблица `user_sessions`

### Назначение

Таблица используется для хранения пользовательских сессий авторизации и управления доступом к платформе. Она необходима для backend API и механизма безопасности приложения.

### Поля и предполагаемые типы

| Поле | Тип | Назначение |
|---|---|---|
| `id` | `uuid` | Первичный ключ сессии. |
| `user_id` | `uuid` | Внешний ключ на таблицу `users`. |
| `session_token` | `varchar(255)` | Токен или идентификатор сессии. |
| `ip_address` | `varchar(64)` | IP-адрес клиента на момент входа. |
| `user_agent` | `text` | Строка user-agent клиента. |
| `expires_at` | `timestamp` | Время истечения сессии. |
| `revoked_at` | `timestamp` | Время принудительного завершения сессии, если применимо. |
| `created_at` | `timestamp` | Момент создания сессии. |
| `updated_at` | `timestamp` | Момент последнего обновления записи. |

### Связи с другими сущностями

- Каждая запись `user_sessions` относится к одному пользователю из `users`.
- Один пользователь может иметь несколько параллельных или исторических сессий.

## Основные связи между таблицами

Ключевые связи логической схемы могут быть представлены следующим образом:

- `sources` -> `news_items`: один источник содержит множество новостных материалов;
- `news_items` -> `events`: одна новость может порождать одно или несколько событий;
- `assets` -> `events`: один актив может быть связан со множеством событий;
- `assets` -> `price_candles`: один актив имеет множество рыночных свечей;
- `assets` -> `technical_indicators`: один актив имеет множество наборов индикаторов;
- `assets` -> `forecasts`: для одного актива может быть сформировано множество прогнозов;
- `assets` -> `market_regime`: для актива может рассчитываться история режимов;
- `users` -> `user_sessions`: один пользователь имеет множество сессий;
- `users` -> `forecasts`: пользователь может иметь связанную историю сохраненных прогнозов;
- `events` -> `forecasts`: событие может участвовать в построении нескольких прогнозов;
- `market_regime` -> `forecasts`: один режим рынка может быть использован при формировании нескольких прогнозов.

## Логика движения данных по схеме

Последовательность заполнения таблиц в рамках работы платформы предполагается следующей:

1. В таблицу `sources` заносятся сведения об используемых внешних источниках.
2. Новостные материалы сохраняются в `news_items`.
3. Из новостей извлекаются структурированные события и записываются в `events`.
4. Рыночные данные по активам загружаются в `price_candles`.
5. На основе временных рядов рассчитываются признаки и сохраняются в `technical_indicators`.
6. С учетом новостей, событий, технических признаков и общего контекста рассчитывается запись в `market_regime`.
7. На основе совокупности факторов сохраняется итоговый прогноз в `forecasts`.
8. Пользовательские операции авторизации фиксируются в `users` и `user_sessions`.

## Ограничения и дальнейшее развитие схемы

Представленная схема является логическим проектным уровнем и не фиксирует все возможные физические особенности реализации, такие как индексы, партиционирование, триггеры и окончательные ограничения целостности. Эти элементы должны быть определены на этапе перехода от проектной документации к реализации базы данных.

При дальнейшем развитии платформы схема может быть расширена следующими сущностями:

- таблицами для хранения макроэкономических рядов;
- таблицами для хранения сырьевых временных рядов;
- таблицами журналирования загрузок и ошибок интеграции;
- таблицами для хранения расширенного состава факторов кризисометра;
- связующими таблицами для реализации отношений многие-ко-многим между событиями и активами.

# AGENTS.md

## Назначение репозитория
Этот репозиторий используется для подготовки бакалаврской выпускной квалификационной работы и разработки программного продукта.

Тема ВКР:
«Разработка облачной платформы анализа и прогнозирования реакции фондового рынка на основе информации из открытых источников с использованием методов искусственного интеллекта».

## Главное правило по структуре ВКР
Использовать трехглавную структуру ВКР, согласованную с научным руководителем, как единственно актуальную, даже если она отличается от типовых методических рекомендаций.

Структура ВКР:
Глава 1. Анализ предметной области и требований к платформе

1.1. Особенности фондового рынка как объекта интеллектуального анализа  
1.2. Классификация новостных, макроэкономических и сырьевых факторов, влияющих на рынок  
1.3. Методы анализа тональности, извлечения событий и прогнозирования реакции рынка на финансовых данных  
1.4. Анализ существующих платформ и инструментов оценки влияния новостей на фондовый рынок  
1.5. Выработка рекомендаций к разработке облачной платформы  
1.6. Функциональные и нефункциональные требования к системе  

Глава 2. Проектирование архитектуры и алгоритмов платформы

2.1. Структурная схема и общая архитектура облачной платформы  
2.2. Проектирование базы данных и структуры обработки данных  
2.3. Проектирование интеллектуального модуля анализа новостей и генерации сигналов  
2.4. Проектирование кризисометра и логики интегральной оценки состояния рынка  
2.5. Проектирование пользовательского интерфейса и API взаимодействия компонентов  

Глава 3. Разработка, тестирование и оценка платформы

3.1. Реализация серверной части платформы на Go  
3.2. Реализация клиентской части платформы на React  
3.3. Реализация AI-модуля обработки новостей и генерации комбинированного сигнала  
3.4. Тестирование на исторических рыночных кейсах  
3.5. Оценка качества прогнозов и эксплуатационных характеристик системы

Не изменять эту структуру без прямого указания пользователя.

## Контекст проекта
Платформа должна:
- собирать открытые новостные, макроэкономические и рыночные данные;
- анализировать события, влияющие на рынок;
- рассчитывать технические и рыночные признаки;
- формировать недельный комбинированный прогноз реакции рынка;
- отображать индекс состояния рынка в виде кризисометра.

## Активы первой версии
- индекс Московской биржи;
- Сбер;
- Лукойл;
- Газпром;
- Яндекс;
- нефть и газ как внешние факторы.

## Горизонт прогноза
Основной горизонт прогнозирования: 1 неделя.

## Формат прогноза
Система должна выдавать комбинированный сигнал, включающий:
- направление;
- силу сигнала;
- уверенность;
- краткое объяснение;
- связь сигнала с событием и рыночным контекстом.

## Технологический стек
- Backend: Go
- Frontend: React
- База данных: PostgreSQL
- AI/NLP: внешний API OpenAI
- Авторизация: email + password
- История прогнозов: обязательна

## Общая архитектура системы
Система включает:
1. модуль сбора данных;
2. модуль предобработки новостей и событий;
3. модуль расчета технических признаков;
4. интеллектуальный модуль анализа новостей и генерации сигнала;
5. модуль расчета кризисометра;
6. backend API;
7. frontend на React;
8. модуль авторизации;
9. хранение истории прогнозов.

## Логика прогнозирования
Итоговый сигнал не должен строиться только на тексте новости.

Перед обращением к AI необходимо учитывать:
- недельный тренд;
- доходность;
- RSI;
- волатильность;
- положение в канале;
- локальные уровни;
- тип события;
- режим рынка;
- влияние нефти и газа;
- макроэкономический контекст.

AI используется для:
- анализа текста новости;
- определения тональности;
- извлечения события;
- объединения текстовых, рыночных и технических признаков;
- генерации итогового прогноза;
- формирования краткого объяснения прогноза.

## Кризисометр
Кризисометр должен быть гибридным интегральным индексом состояния рынка и учитывать:
- рыночный стресс;
- новостный стресс;
- макроэкономический стресс;
- сырьевой фон;
- ширину рынка и поведение ключевых активов.

## Правила работы с источниками
Перед разработкой любого модуля:
1. проверять актуальные официальные источники данных;
2. приоритет отдавать официальным API, биржам, регуляторам, статистическим службам и центрам раскрытия информации;
3. все релевантные источники добавлять в `04_materials/sources_registry.md`;
4. для каждого источника фиксировать:
   - название,
   - URL,
   - тип данных,
   - способ доступа,
   - периодичность обновления,
   - статус,
   - дату последней проверки;
5. не использовать сомнительные неофициальные источники как единственный источник.
